# YieldVitals 網路測速問題 - 研究總結

## 🎯 核心問題（一句話總結）
**網路測試完全依賴單一的 Cloudflare 服務，在緩存、超時、故障轉移、數據驗證等多個環節存在缺陷，導致在特定網路環境下測不出速度或結果不准確。**

---

## 🔴 找出的 6 大問題

### 1. **單一依賴 + CORS 限制** 
- 只依賴 `https://speed.cloudflare.com`，沒有備用方案
- 企業代理、地區限制、CDN 故障時全部失敗

### 2. **超時設置固定不合理**
- 下載 20s、上傳 8s 的固定超時
- 高延遲網路（>500ms）或慢速網路（<10Mbps）直接超時失敗

### 3. **防緩存機制不足**
- `t=${Date.now()}` 精度不夠，快速重試時會重複
- 某些代理和瀏覽器仍會緩存

### 4. **數據完整性未驗證**
- 下載測試只檢查 HTTP 狀態，不驗證字節完整性
- 可能只收到 50% 的數據但仍認為成功

### 5. **單次測試導致波動大**
- 只運行 1 次，無法平均化網路波動
- 結果波動幅度大（±50%）

### 6. **沒有降級和診斷方案**
- 失敗時只返回 0，不知道原因
- 用戶無法區分「網路慢」還是「測試故障」

---

## ✅ 改進方案（3 個優先級）

### 🟢 優先級 1：快速修復（推薦，30 分鐘）
**改進現有 `runNetworkTest` 函數**：

1. **防緩存**：多重時間戳 + 隨機數 + 標頭
   ```javascript
   const noCacheParam = `t=${Date.now()}_r${Math.random()}_${crypto.getRandomValues(...)}`;
   ```

2. **動態超時**：根據 Ping 自動調整
   ```javascript
   - Ping >500ms：用 5MB 文件，45s 超時
   - Ping 100-500ms：用 25MB 文件，30s 超時
   - Ping <100ms：用 100MB 文件，20s 超時
   ```

3. **數據驗證**：檢查 Content-Length 和完整性
   ```javascript
   const completionRate = receivedLength / expectedSize;
   if (completionRate < 0.9) warn('不完整');
   ```

4. **多次迭代**：從 1 次改為 3 次，計算中位數

### 🟡 優先級 2：可靠性增強（1-2 天）
**添加故障轉移和備用方案**：

1. **備用端點列表**
   ```javascript
   - Cloudflare (主)
   - Cloudflare US (備用)
   - 自建本地服務 (降級)
   ```

2. **自動故障轉移**：端點不可用時自動切換

3. **本地測試服務**：提供 `/api/network/ping`, `/api/network/download`, `/api/network/upload`

### 🔵 優先級 3：優化和監控
**長期改進**：

1. **網路分數納入總分**（目前被排除）
2. **詳細診斷報告**（顯示為什麼失敗）
3. **實時監控面板**（全球端點可用性）

### 防緩存機制（最重要）

```javascript
// 改進：t=${Date.now()}_r${Math.random()}_${crypto.getRandomValues(...)}
// 效果：URL 每次都不同，完全避免緩存
```

### 動態適應（針對各類網路）
```javascript
// 根據 Ping 測量，動態選擇：
// - 文件大小（5MB / 25MB / 100MB）
// - 超時時間（45s / 30s / 20s）
// 效果：支持 1Mbps 到 1Gbps 的全範圍網路
```

```javascript
// 驗證接收的字節是否完整
const completionRate = receivedLength / expectedSize;

}
```
